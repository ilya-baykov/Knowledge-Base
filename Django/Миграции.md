# 📋 Шпаргалка по миграциям Django

---

## 🔧 Флаги команды makemigrations

```bash
python manage.py makemigrations references --empty --name=add_disputed_status
```

| Часть команды | Что делает |
|---|---|
| `makemigrations` | Создаёт файл миграции |
| `references` | Указывает **конкретное приложение** (app). Без него Django создаст миграции для всех приложений сразу |
| `--empty` | Создаёт **пустую миграцию** — без автоматически обнаруженных изменений схемы. Нужно для data migrations, когда ты хочешь сам написать логику |
| `--name=add_disputed_status` | Даёт миграции **человекочитаемое имя**. Итоговый файл будет `0005_add_disputed_status.py` вместо `0005_auto_20240101_1234.py` |

> 💡 **Всегда используй `--name`** — через полгода ты скажешь себе спасибо, когда будешь искать нужную миграцию в списке из 40 файлов.

---

## 📁 Структура файла миграции

```python
# Каждый файл миграции — это класс Migration с двумя обязательными атрибутами

class Migration(migrations.Migration):

    # dependencies — список миграций, которые ДОЛЖНЫ выполниться ДО этой.
    # Django строит граф зависимостей и выполняет миграции в правильном порядке.
    # Формат: ("имя_приложения", "имя_файла_без_.py")
    dependencies = [
        ("references", "0001_initial"),
    ]

    # operations — список действий, которые выполняет эта миграция.
    # Каждая операция умеет применяться (forward) и откатываться (backward).
    operations = [
        # Здесь будут операции...
    ]
```

## 🗄️ Случай 1: Начальное наполнение справочника (data migration)

**Когда использовать:** при первом развёртывании — чтобы сразу заполнить таблицу нужными данными.

```bash
# Создаём пустую миграцию специально для данных
python manage.py makemigrations references --empty --name=populate_stash_statuses
```

```python
from django.db import migrations

# Данные выносим в константу — так их легко читать и редактировать
STATUSES = [
    {"order": 1, "slug": "draft",    "name": "Черновик",     "description": ""},
    {"order": 2, "slug": "active",   "name": "Активный",     "description": ""},
    {"order": 3, "slug": "sold",     "name": "Куплен",       "description": ""},
    {"order": 4, "slug": "found",    "name": "Найден",       "description": ""},
    {"order": 5, "slug": "disputed", "name": "Оспаривается", "description": ""},
    {"order": 6, "slug": "archived", "name": "Архивный",     "description": ""},
]


def populate_stash_statuses(apps, schema_editor):
    """
    Forward-функция: применяется при migrate.

    ВАЖНО: используй apps.get_model(), а НЕ прямой импорт модели!

    Причина: миграции выполняются в историческом контексте.
    Если ты импортируешь модель напрямую — получаешь её ТЕКУЩУЮ версию,
    а не ту, которая была актуальна в момент написания этой миграции.
    Это может сломать миграцию при будущих изменениях модели.
    """
    StashStatus = apps.get_model("references", "StashStatus")

    for data in STATUSES:
        # get_or_create — идемпотентная операция:
        # если запись уже есть — просто пропускаем, не падаем с ошибкой.
        # slug используем как уникальный ключ поиска.
        StashStatus.objects.get_or_create(slug=data["slug"], defaults=data)


def depopulate_stash_statuses(apps, schema_editor):
    """
    Backward-функция: вызывается при migrate --fake или rollback.

    Всегда пиши обратную функцию — это хороший тон.
    Она позволяет откатить миграцию командой:
        python manage.py migrate references 0001
    """
    StashStatus = apps.get_model("references", "StashStatus")
    slugs = [d["slug"] for d in STATUSES]
    StashStatus.objects.filter(slug__in=slugs).delete()


class Migration(migrations.Migration):

    dependencies = [
        # Указываем предыдущую миграцию этого приложения
        ("references", "0001_initial"),
    ]

    operations = [
        # RunPython принимает два аргумента:
        # 1. forward-функция (apply)
        # 2. backward-функция (unapply / rollback)
        migrations.RunPython(populate_stash_statuses, depopulate_stash_statuses),
    ]
```

## ➕ Случай 2: Добавление новой записи в справочник

**Когда использовать:** когда бизнес-логика расширилась и нужен новый статус.

> ⚠️ **Золотое правило: никогда не редактируй старые миграции!**
> Создавай новую миграцию для каждого изменения данных.

```bash
python manage.py makemigrations references --empty --name=add_disputed_status
```

```python
from django.db import migrations


def add_disputed(apps, schema_editor):
    StashStatus = apps.get_model("references", "StashStatus")
    StashStatus.objects.get_or_create(
        slug="disputed",
        defaults={
            "order": 5,
            "name": "Оспаривается",
            "description": "Статус оспаривается одной из сторон",
            "is_active": True,
        }
    )


def remove_disputed(apps, schema_editor):
    StashStatus = apps.get_model("references", "StashStatus")
    StashStatus.objects.filter(slug="disputed").delete()


class Migration(migrations.Migration):

    dependencies = [
        # Зависим от миграции, которая наполнила таблицу впервые
        ("references", "0002_populate_stash_statuses"),
    ]

    operations = [
        migrations.RunPython(add_disputed, remove_disputed),
    ]
```

## ✏️ Случай 3: Изменение схемы — добавление нового поля

**Когда использовать:** когда добавляешь новое поле в модель.

```python
# В models.py добавляем поле:
color = models.CharField(max_length=7, default="#000000", help_text="HEX-цвет для UI")
```

```bash
# Django сам обнаружит изменение и сгенерирует миграцию схемы
python manage.py makemigrations references --name=add_color_to_stash_status
```

Django автоматически создаст:

```python
class Migration(migrations.Migration):

    dependencies = [
        ("references", "0003_add_disputed_status"),
    ]

    operations = [
        # AddField — стандартная операция изменения схемы.
        # Django генерирует её сам — руками писать не нужно.
        migrations.AddField(
            model_name="stashstatus",
            name="color",
            field=models.CharField(default="#000000", max_length=7),
        ),
    ]
```

## 🔄 Случай 4: Изменение данных при изменении схемы

**Когда использовать:** добавил поле и хочешь сразу проставить значения для существующих записей.

```bash
# Сначала Django генерирует миграцию схемы
python manage.py makemigrations references --name=add_color_to_stash_status

# Затем создаём отдельную миграцию для заполнения данных
python manage.py makemigrations references --empty --name=fill_colors_for_stash_statuses
```

```python
from django.db import migrations

# Словарь slug → цвет
COLORS = {
    "draft":    "#9E9E9E",  # серый
    "active":   "#4CAF50",  # зелёный
    "sold":     "#2196F3",  # синий
    "found":    "#FF9800",  # оранжевый
    "disputed": "#F44336",  # красный
    "archived": "#607D8B",  # сине-серый
}


def fill_colors(apps, schema_editor):
    StashStatus = apps.get_model("references", "StashStatus")
    for slug, color in COLORS.items():
        # update() эффективнее, чем get() + save() — делает один SQL запрос
        StashStatus.objects.filter(slug=slug).update(color=color)


def clear_colors(apps, schema_editor):
    StashStatus = apps.get_model("references", "StashStatus")
    StashStatus.objects.all().update(color="#000000")


class Migration(migrations.Migration):

    dependencies = [
        # ВАЖНО: зависим от миграции схемы — поле уже должно существовать!
        ("references", "0004_add_color_to_stash_status"),
    ]

    operations = [
        migrations.RunPython(fill_colors, clear_colors),
    ]
```

## 🗑️ Случай 5: Удаление поля из модели

```python
# Убираем поле из models.py, затем:
python manage.py makemigrations references --name=remove_color_from_stash_status
```

> ⚠️ **Осторожно на продакшене!** Если код уже задеплоен и обращается к этому полю,
> сначала убери обращения к полю в коде, задеплой, и только потом удаляй поле миграцией.

## 🚀 Полезные команды

```bash
# Применить все миграции
python manage.py migrate

# Применить миграции только одного приложения
python manage.py migrate references

# Откатить до конкретной миграции (включительно)
python manage.py migrate references 0002

# Откатить ВСЕ миграции приложения (до состояния "нет таблиц")
python manage.py migrate references zero

# Посмотреть список всех миграций и их статус (X = применена)
python manage.py showmigrations

# Посмотреть SQL, который выполнит миграция — без применения
python manage.py sqlmigrate references 0003

# Пометить миграцию как применённую БЕЗ реального выполнения
# Полезно когда таблица уже существует (например, на legacy-базе)
python manage.py migrate references 0003 --fake

# Пометить только начальную миграцию как fake (частый кейс для legacy)
python manage.py migrate references --fake-initial
```

## 🔍 Проверка перед деплоем

```bash
# Убеждаемся, что все миграции созданы и ничего не забыто
# Если вывод пустой — всё хорошо
python manage.py makemigrations --check

# Можно также явно проверить конкретное приложение
python manage.py makemigrations references --check
```

## 🧩 Зависимости между приложениями

Если модели из разных приложений ссылаются друг на друга через ForeignKey —
миграции должны явно указывать зависимости:

```python
class Migration(migrations.Migration):

    dependencies = [
        # Зависим от миграции другого приложения,
        # потому что наша таблица ссылается на его таблицу через ForeignKey
        ("references", "0001_initial"),
        ("stashes", "0003_add_status_field"),
    ]
```

## ❌ Частые ошибки

| Ошибка | Почему плохо | Как правильно |
|---|---|---|
| Импортировать модель напрямую в миграции | Миграция сломается при изменении модели в будущем | Использовать `apps.get_model()` |
| Редактировать старые миграции | Ломает историю у всех разработчиков и на всех серверах | Создавать новую миграцию |
| Не писать backward-функцию | Нельзя откатить миграцию | Всегда писать функцию отката |
| Коммитить без миграций | На сервере схема не совпадёт с моделями | Запускать `makemigrations --check` в CI |
| Удалять файлы миграций вручную | Django теряет историю состояния схемы | Никогда не удалять примёненные миграции |

---

## 💡 Главные принципы

1. **Одна миграция — одно изменение.** Не мешай в одну миграцию изменение схемы и данных.
2. **Миграции — это история.** Не редактируй прошлое, только добавляй новое.
3. **`apps.get_model()` — всегда.** Никогда не импортируй модели напрямую внутри функций миграции.
4. **Пиши backward-функцию.** Даже если сейчас не планируешь откат — он может понадобиться.
5. **Давай имена миграциям.** `--name=` спасает при отладке и code review.
6. **Коммить миграции в git.** Файлы миграций — часть кодовой базы, не артефакты сборки.
