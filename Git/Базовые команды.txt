  ⚙️  ПЕРВОНАЧАЛЬНАЯ НАСТРОЙКА
git config --global user.name 'Имя'	Установить имя (один раз)
git config --global user.email 'e@mail.com'	Установить email
git config --global core.editor nano	Редактор для сообщений коммита
git config --list	Посмотреть все настройки

  📁  СОЗДАНИЕ И КЛОНИРОВАНИЕ
git init	Инициализировать репозиторий в текущей папке
git clone <url>	Клонировать репозиторий
git clone <url> my-folder	Клонировать в папку my-folder

  🔍  СТАТУС И ПРОСМОТР ИЗМЕНЕНИЙ
git status	Что изменено, что в стейдже, что нет
git diff	Что изменено, но НЕ добавлено в стейдж
git diff --staged	Что добавлено в стейдж и готово к коммиту
git log --oneline	Краткая история коммитов
git log --oneline --graph	История с деревом веток
git show <commit>	Посмотреть конкретный коммит

  ✅  СТЕЙДЖИНГ И КОММИТЫ
git add file.py	Добавить файл в стейдж
git add .	Добавить ВСЕ изменения в стейдж
git add -p	Интерактивно выбрать части файла
git commit -m 'feat: add login'	Закоммитить с сообщением
git commit --amend	Изменить последний коммит (до push!)
git reset HEAD file.py	Убрать файл из стейджа (сохранив изменения)
git restore file.py	Откатить изменения в файле (опасно!)
💡 Формат сообщений: feat/fix/docs/refactor/test: краткое описание

  🌿  РАБОТА С ВЕТКАМИ
git branch	Список веток (текущая помечена *)
git branch feature/my-task	Создать ветку
git checkout feature/my-task	Переключиться на ветку
git checkout -b feature/my-task	Создать и сразу переключиться
git switch -c feature/my-task	То же самое (новый синтаксис)
git branch -d feature/my-task	Удалить ветку (после мёрджа)
git branch -D feature/my-task	Удалить ветку принудительно
git merge feature/my-task	Влить ветку в текущую
git rebase develop	Rebas текущей ветки на develop

  🌐  УДАЛЁННЫЙ РЕПОЗИТОРИЙ (REMOTE)
git remote -v	Показать удалённые репозитории
git fetch origin	Скачать изменения (без применения)
git pull origin develop	Fetch + merge: обновить ветку
git pull --rebase origin develop	Fetch + rebase (чище история)
git push origin feature/my-task	Запушить ветку на remote
git push -u origin feature/my-task	Запушить и установить upstream
git push --force-with-lease	Force push (безопасный вариант)
💡 pull --rebase предпочтителен в командах — история остаётся линейной

  📦  STASH — ВРЕМЕННОЕ ХРАНИЛИЩЕ
git stash	Спрятать незакоммиченные изменения
git stash pop	Вернуть последний stash
git stash list	Список всех stash
git stash apply stash@{1}	Применить конкретный stash
git stash drop stash@{0}	Удалить stash
💡 Stash — спасение, когда надо срочно переключиться на другую задачу

  🔙  ОТМЕНА И ИСПРАВЛЕНИЕ ОШИБОК
git revert <commit>	Создать коммит, отменяющий изменения (безопасно!)
git reset --soft HEAD~1	Отменить коммит, сохранив изменения в стейдже
git reset --mixed HEAD~1	Отменить коммит, изменения остаются в файлах
git reset --hard HEAD~1	Отменить коммит + удалить изменения (ОПАСНО)
git cherry-pick <commit>	Перенести конкретный коммит в текущую ветку
💡 revert — безопасен на shared-ветках. reset --hard необратим, осторожно!

  🔀  GIT FLOW — ТИПЫ ВЕТОК

Ветка	Пример	Назначение
main / master	main	Прод-код. Только через PR, никогда напрямую!
develop	develop	Основная ветка разработки. Сюда вливают все фичи.
feature/...	feature/user-auth	Новая функциональность. Ветвится от develop.
bugfix/...	bugfix/login-error	Исправление бага в разработке. От develop.
hotfix/...	hotfix/critical-crash	Срочный фикс прода. От main, мёрджится в main и develop.
release/...	release/1.2.0	Подготовка к релизу. Только фиксы, не фичи.

  📋  GIT FLOW — ТИПИЧНЫЙ РАБОЧИЙ ПРОЦЕСС

1	Получить задачу	Берёшь тикет из Jira/Linear/etc. Пример: PROJ-123
2	Создать ветку	git checkout develop && git pull git checkout -b feature/PROJ-123-user-auth
3	Разработка	Пишешь код, делаешь небольшие атомарные коммиты
4	Push	git push -u origin feature/PROJ-123-user-auth
5	Pull Request	Открываешь PR: feature/... → develop. Описываешь что сделал.
6	Code Review	Коллеги смотрят, оставляют комментарии, ты фиксишь
7	Merge	После апрува — мёрдж в develop (обычно Squash merge)
8	Удалить ветку	git branch -d feature/PROJ-123-user-auth

  💎  СОВЕТЫ ДЛЯ РАБОТЫ В КОМАНДЕ
Мелкие коммиты	Коммить часто и по одной логической единице
Хорошие сообщения	feat: add user auth — понятно что и зачем
Pull перед push	Всегда делай git pull перед началом работы
Никогда в main/master	Работай только через ветки и PR
.gitignore	Добавляй __pycache__, .env, .venv, *.pyc
Squash on merge	Команды часто сквошат коммиты PR в один
Protected branches	main и develop обычно защищены — нельзя пушить напрямую
